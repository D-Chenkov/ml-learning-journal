What is Kubernetes? Defined as Open Source, a container orchestration tool. (Curious: I thought what we read previously said it did NOT need orchestration.) Manages containerized applications in DIFFERENT ENVIRONMENTS (unsure if at the same time).

What does it solve? Hundreds of containers being managed isn't simple.
What does it offer? High availability, scalability, disaster recovery.

COMPONENTS
Pod: smallest unit of k8s, an abstraction over a container. Usually run 1 application per pod (DB / app / side service etc). We only interact with the kubernetes layer, don't need to touch the container directly. Each pod gets its own IP address, BUT a new IP is assigned on re-creation.

Service: a permanent IP address attached to a pod. The lifecycle of pod and service are NOT connected. External Service vs Internal Service: created on demand by us depending on whether it is client-facing or just internal.

Ingress: don't want to talk to 2194.214.24.24:8008 directly? Use Ingress, so the request goes to Ingress which forwards to the appropriate service.

ConfigMap and Secrets: the DB URL is usually built into the application. ConfigMap is the external configuration of your app, e.g. DB_URL = mongo-db-service. NEVER put credentials in a ConfigMap. That's what Secret is for. Secrets are base64 encoded, but the built-in security mechanism is not enabled by default.

Volumes: data storage. If data is generated anywhere (a DB for example), it can be attached to a volume, local or remote. K8s does NOT manage data persistence.

Deployment / StatefulSet: if a pod dies we'd have downtime, so within nodes we keep replicas connected to the same service. Correction: you don't make nodes/pods willy-nilly, you create a blueprint (DEPLOYMENT) for the pods. You don't make pods manually, you make Deployments that scale and ensure the service. A DB can't be replicated via a Deployment (data inconsistencies). StatefulSet is for STATEFUL apps (databases).
DEPLOYMENT = STATELESS apps. STATEFULSET = STATEFUL apps. StatefulSets are NOT easy, so DBs are often hosted outside the K8s cluster.

KUBERNETES ARCHITECTURE
Nodes have multiple application pods running on them. 3 processes must be installed on every worker node; worker nodes do the actual work.
- Container Runtime
- Kubelet: interacts with both the container and the node; starts the pod with a container inside.
- Kube Proxy: forwards requests; lightweight, low overhead.
How to interact with the cluster, schedule, monitor, reschedule/restart pods, join new nodes? All through the MASTER NODE.

Master Processes (4 run on every master node):
- API Server: update or query; the cluster gateway and gatekeeper for authentication.
- Scheduler: to schedule a new pod, request goes API Server -> Scheduler -> decides where to put the pod, validating available resources and balancing so it lands where least used. The Scheduler ONLY decides which node a new pod goes to.
- Controller Manager: detects cluster state changes; Controller Manager -> Scheduler -> decides again based on resource calculation.
- Etcd: key-value store of the cluster state (the cluster brain). Any change gets saved here. Lets it determine: is the cluster healthy, what resources are available, did the cluster state change. DOES NOT save application data.

So a Kubernetes cluster has MULTIPLE master nodes, each API server load balanced, and etcd as distributed storage across all master nodes.

MINIKUBE and KUBECTL
Normally master and worker are separate machines. Can you test locally?
Minikube: a 1-node cluster where master and node run on the same machine. Docker pre-installed. Runs using a hypervisor (VirtualBox, etc).
Kubectl: command-line tool to interact with a k8s cluster, via the API Server. Interact via UI, API, or CLI (kubectl). Worker processes let pods run on nodes: create pods, destroy pods, create services. Kubectl works for ANY cluster, not just the local one-node setup.

MINIKUBE INSTALLATION / COMMANDS
- minikube start --vm-driver=hyperkit : start the cluster (DOES NOT WORK ON ARM64).
- kubectl get nodes : status of the nodes.
- minikube status : overall status of host, kubelet, apiserver, kubeconfig.
- kubectl version : client/server versions.
Kubectl CLI = for configuring the cluster; Minikube CLI = for starting up / deleting the cluster.

MAIN KUBECTL COMMANDS
CRUD: kubectl create deployment [name] / kubectl edit deployment [name] / kubectl delete deployment [name].
Status: kubectl get nodes | pod | services | replicaset | deployment.
Debugging: kubectl logs [pod name] (logs to console); kubectl exec -it [pod name] -- bin/bash (interactive terminal).

REMINDER: we create DEPLOYMENTS, an abstraction over pods. A ReplicaSet manages the replicas of the pod; we don't control replicasets/replicas directly.
LAYERS: Deployment manages a ReplicaSet, which manages a Pod, which is an abstraction over a Container. Everything below a Deployment is managed by Kubernetes.
Edit provides an auto-generated config file with default values (metadata / spec / etc). kubectl describe pod [name] for details. kubectl apply -f [file].yaml applies the config file.

K8S YAML CONFIGURATION FILE (3 parts)
- Metadata: name, labels.
- Specification (spec): replicas, selector, ports, etc, and a Template (a config inside the config that applies to a pod, the pod blueprint).
- Status: auto-generated and edited by Kubernetes; the desired vs actual state, updated constantly, data sourced from etcd (the cluster brain).
Attributes of spec are SPECIFIC to the KIND (e.g. Deployment, Service).

CONNECTING COMPONENTS
Connections are established by labels and selectors. Metadata holds the labels; spec holds the selectors. Any key-value pair works. Pods get their label via the template blueprint, and the selector matches that label. The Deployment has a label too, and the Service selector uses it to connect the service to the deployment/pods. DB Service -> nginx Service -> Pod.

**Confused by:**
- Service -> pods: a Service targets pods by LABEL SELECTOR (and load-balances across the matched set), not by the replica count. Keep the two separate: Deployment governs HOW MANY pods; Service selector governs WHICH pods to route to. Membership is dynamic (whatever currently matches the selector).
- StatefulSet: what it provides over a Deployment for databases = (1) stable, unique network identity (ordinal names/DNS that persist), (2) stable per-pod persistent storage (each pod reattaches to its own volume), (3) ordered create/scale/delete. Caveat: it does NOT replicate data for you; you still configure the DB's own primary/replica replication. Related DB concept = ACID (Atomicity, Consistency, Isolation, Durability).
- Self-healing chain (mend closed): ReplicaSet controller (in kube-controller-manager) notices actual < desired and creates a pod -> Scheduler picks the node -> kubelet on that node starts the container (control plane decides, node executes). Service routes only to ready pods, so traffic rebalances to survivors meanwhile.
- Secret detail: base64 is ENCODING, not encryption (trivially reversible). Real protection needs encryption at rest for etcd, tight RBAC, and ideally an external secrets manager (Vault, GCP Secret Manager).
