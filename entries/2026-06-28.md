**Learned (from the reading):**
- Validate incoming data on four axes: file format/schema consistency, overall data distribution (row counts, null counts), feature distribution, and statistical checks. Anomalies often mean corrupt, duplicated, or tampered data.
- Monitor models when retraining: keep a recent holdout (e.g., the latest 7+ days) and validate against it before deploying. Shadow releasing runs the new model on live traffic without serving its output, to compare metrics safely.
- Interpretability tools (LIME, SHAP): a feature that suddenly dominates when it did not before can signal a data leak.
- Simplify engineering: log everything (MLflow is a good fit), serve via containers (ties to this week's Docker work), keep rollbacks easy. KISS: complexity creeps in, beat it back; generalize first, personalize later.
- Google's Rules of ML (referenced): use positional features to avoid feedback loops, learn thresholds from data, audit a sample of your data periodically.

**Concepts reinforced:**
- Cache ordering (old mend item): requirements on top (least-changing), code on bottom (most-changing); CACHED on the pip layer when only train.py changes.
- Image (immutable layers, from build) vs container (image + writable layer, from run). The saved model.joblib lives in the container's writable layer and is lost on --rm unless a volume is mounted.
- Registry hierarchy made concrete: registry (Docker Hub) contains repository (dockerid/iris-classifier) contains tagged image (:v1). The docker tag step is needed because the local name has no registry/namespace; push requires the fully qualified name. Deleting local copies before the pull is the real verification the push worked.

**Confused by:**
- Shadow releasing (priority): its outputs are NOT served to users, which is exactly why it is safe. It runs the new model on live traffic in parallel and logs/compares, with zero user exposure. I had it serving live (that is a canary, not a shadow). Cost is double compute and no real outcome signal (no one acts on the shadow output).
- Valid Palindrome space: the O(n) extra space in my first attempt was the new string re.sub allocates (plus the reversed copy). The two-pointer version allocates only two integer indices = O(1). isalnum() is what lets me skip junk inline so I never build the cleaned string.
- Docker cache nuance: the RUN pip install layer is cached on prior layers + command + requirements.txt, NOT on PyPI's current state. An unpinned upstream change does not bust the cache, which is why pinning matters.
