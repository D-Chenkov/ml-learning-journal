HELM TEMPLATING: Deployment/Service configs are near-identical across environments (only image/version tags differ). Without Helm you keep separate near-duplicate files; with Helm you (1) define a common blueprint (template) and (2) replace dynamic values with placeholders. A template YAML uses placeholders like {{ .Values.name }}, filled from values.yaml (or the --set flag). Benefits: many YAMLs collapse to one templated set, great for CI/CD (inject values on the fly), and you can deploy the same app across dev/staging/prod by packaging your own chart. Chart directory: mychart/ (chart name) with Chart.yaml (metadata), values.yaml (values for templates), charts/ (dependencies), templates/ (the template files filled from values.yaml), plus optional files (README). helm install <chart>. Override with helm install --values=my-values.yaml <chart> (values MERGE/override, not replace) or --set. Release management: Helm 2 had a client (helm CLI) + server (Tiller); Tiller had too many cluster permissions (security issue), so Helm 3 removed it.

K8S VOLUMES: no data persistence out of the box. Requirements: storage independent of the pod lifecycle, available on all nodes, and surviving a cluster crash. Three pieces:
- PersistentVolume (PV): a cluster resource (kind PersistentVolume) that is the INTERFACE to actual physical storage (local disk, NFS, one of 25+ backends). Must exist before the pod that needs it. Admin creates/manages.
- PersistentVolumeClaim (PVC): the REQUEST a pod makes for storage; K8s binds it to a suitable PV. Referenced in the pod config and must be in the same namespace as the pod. The volume is then mounted into the pod, then into the container.
- StorageClass (SC): DYNAMIC provisioning - creates a PV on demand when a PVC claims it, via a "provisioner" attribute (internal kubernetes.io or external). Abstracts the storage provider and its parameters.
Roles: admin provisions storage + PVs (or a StorageClass); users claim via PVC. ConfigMap and Secret are also volume types (local volumes managed by K8s, not via PV/PVC): create the ConfigMap/Secret, then mount it into the pod as a directory of data (config file, certificate).

K8S STATEFULSET: stateful apps (databases, apps that store data) vs stateless (no state, each request is new). Stateless -> Deployment: identical, interchangeable pods, random order/hashes, one service load-balances to any pod. Stateful -> StatefulSet: pods can't be created/deleted simultaneously, can't be randomly addressed, are NOT identical (pod identity), and keep a sticky identity persistent across rescheduling. Why: with data replicas you'd get inconsistency, so one master handles reads+writes and workers only read, syncing continuously (continuous sync could theoretically allow no PV, but that is dangerous). Two characteristics: predictable pod name (mysql-0) and fixed individual DNS name (mysql-0.svc). K8s helps but gives nothing out of the box: you configure cloning/data sync, make remote storage available, and manage backups yourself. Stateless is a perfect fit for containers; stateful is workable but needs outside help (often a managed DB outside the cluster).

K8S SERVICES (refresher flagged for later): a Service provides a stable IP, load balancing, loose coupling, and works within and outside the cluster. Types:
- ClusterIP: default/most common, internal IP from the node range. Knows which pods to forward to via SELECTORS matching pod labels; knows which port via the port spec.
- Headless: clusterIP set to None, no load balancing / no single virtual IP, so a DNS lookup returns the individual pod IP - lets you address a SPECIFIC pod. Use case: stateful apps like databases.
- NodePort: extension of ClusterIP; exposes on each node's port. Not for production external access, at most browser testing.
- LoadBalancer: extension of NodePort; external access via a cloud load balancer.

**Confused by (minor, now clarified):**
- PV vs PVC labels: PV = the storage resource / interface to physical storage; PVC = the request that binds to a PV. (Pod -> PVC -> bound PV -> physical storage.)
- Service types as a progression: ClusterIP (internal) is the base; NodePort adds a per-node external port on top; LoadBalancer adds a cloud load balancer on top. Headless (clusterIP None) is the odd one out for addressing specific pods.
