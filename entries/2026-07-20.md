*Minimum Window Substring (two rounds of debugging, own counterexamples):*
- First attempt used a fixed `windowSize = len(t)` as the trigger for when to check/shrink the window. That's structurally wrong: the minimum valid window length isn't known in advance and can be longer than `len(t)`.
  - Round 1 bug: `end - start >= windowSize` and `s[start:end]` both had off-by-one errors (window length is `end - start + 1`, inclusive of both ends). Fixing just those two happened to fix `"ADOBECODEBANC"/"ABC"` -> `"BANC"`, but only by accident: the two off-by-ones were partially canceling each other out.
  - Round 2 (real) bug, found via my own counterexample `"OUZODYXAZV"/"XYZ"` expecting `"YXAZ"`: shrinking the window unconditionally once it hits a length threshold, regardless of validity, evicts characters the window will still need later (in this case `Y` gets shrunk out before `Z` reappears). The correct approach shrinks only while the window is *valid*, in a `while` loop, recording the best answer at each valid state before continuing to shrink, and lets the window grow arbitrarily large before validity is first reached. Length is not the reliable signal; validity is.
  - Also replaced the per-iteration `Counter` subtraction (`sub_counter - windowCount`) with a running `have`/`needed` counter incremented/decremented as individual character counts cross their required thresholds: an O(1) incremental update instead of an O(k) recomputation every step.
  - Final structure (have/needed validity, while-shrink):
    ```python
    def minWindow(self, s: str, t: str) -> str:
        if not t or not s:
            return ""
        need = Counter(t)
        window = Counter()
        have, needed = 0, len(need)
        start = 0
        best = (float("inf"), 0, 0)
        for end, char in enumerate(s):
            window[char] += 1
            if char in need and window[char] == need[char]:
                have += 1
            while have == needed:
                if end - start + 1 < best[0]:
                    best = (end - start + 1, start, end)
                window[s[start]] -= 1
                if s[start] in need and window[s[start]] < need[s[start]]:
                    have -= 1
                start += 1
        return "" if best[0] == float("inf") else s[best[1]:best[2] + 1]
    ```

*Stanford CS224N: NLP with Deep Learning, Winter 2021, Lecture 1 — Intro & Word Vectors (Christopher Manning):*
- Frames the core problem of NLP: how do you represent word *meaning* in a way a computer can compute with? Opens with human language as a comparatively recent, uniquely powerful communication/cognition system, and ties it to why systems like Google Translate and GPT are notable, they're operating on top of some representation of meaning.
- Traditional approach: **WordNet**, a hand-built lexical database organizing words into synonym sets ("synsets") with hypernym/hyponym (is-a) relationships. Useful and widely used, but Manning walks through its real limitations, not just "it's imperfect":
  - Human-curated and maintained, so it lags behind new words, slang, and shifting usage (my example: "bussin" won't be in it).
  - Missing nuance: it captures discrete synonym membership, not degree of similarity, so it can't say "close but not identical" the way a continuous score could.
  - Sparse for rare and domain-specific terms since it depends on people having gotten around to encoding them.
  - Still has real gaps and inconsistencies in its synonym sets even for common words.
- This motivates the pivot the rest of the course takes: **distributional semantics** ("a word's meaning is given by the words that frequently appear close to it", attributed to the distributional hypothesis) and representing words as dense vectors (**word embeddings**) learned from large corpora, with **Word2Vec** as the specific model introduced as the alternative to hand-built resources like WordNet. Embeddings give continuous similarity (cosine distance between vectors) instead of WordNet's discrete synonym-set membership, and they're learned automatically from text rather than hand-curated, directly addressing the "human-made, doesn't update" limitation.
