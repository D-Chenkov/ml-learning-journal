**Learned (from the K8s concepts docs):**
- Kubernetes is a platform for running distributed systems resiliently; it handles scaling, failover, deployments, and more.
- Capabilities (per the docs): service discovery and load balancing, storage orchestration, automated rollouts and rollbacks, automatic bin packing, self-healing, secret and configuration management, batch execution, horizontal scaling, IPv4/IPv6 dual stack, designed for extensibility.
- What Kubernetes is NOT: not a PaaS. It is container orchestration, but via independent control loops that continuously reconcile actual state to desired state (declarative, self-correcting), which is why the docs say it goes beyond traditional workflow orchestration.
- Cluster = control plane (the brain: global decisions, responds to events) + nodes (the workers: run the pods).
- Control plane: kube-apiserver (the API front door), etcd (key-value store, single source of truth for cluster state), kube-scheduler (assigns pods to nodes), kube-controller-manager (runs the reconciliation controllers), cloud-controller-manager (optional; integrates with cloud provider APIs - the K8s/GCP seam on GKE, relevant for my GCP cert).
- Node components: kubelet (keeps the pod's containers running), kube-proxy (optional; networking rules), container runtime (runs containers).
- Addons: DNS, Web UI (Dashboard), container resource monitoring, cluster-level logging.

Kubernetes, yay. Week 2. I already have some experience with Docker and K8s, so this is a good refresher as I work toward my GCP ML cert.

As the K8s docs describe it, Kubernetes is the system that allows for easily and resiliently running distributed systems. It takes care of all the hard parts: scaling, failures, deployments, and so on. Per their own docs, it provides (in no particular order): service discovery and load balancing, storage orchestration, automated rollouts and rollbacks, automatic bin packing, self-healing, secret and configuration management, batch execution, horizontal scaling, IPv4/IPv6 dual stack, and is designed for extensibility.

Then it covers what Kubernetes is not. Surprisingly, it is not a Platform as a Service; it is simply an orchestration system. An orchestration system that technically eliminates the need for orchestration: rather than running a defined workflow, it uses independent control loops that continuously drive the current state toward the desired state.

Everything we read about Docker is still applicable here, since Kubernetes is built to run containers, so all the benefits of containers carry over. Kubernetes is the layer that organizes all of those containers in a way that serves our goals.

A Kubernetes cluster is made of a control plane and the nodes that get fed to the API of that control plane. The control plane components are the kube-apiserver, etcd, kube-scheduler, kube-controller-manager, and the optional cloud-controller-manager. The node components run on every node and maintain the running pods: the kubelet, the (optional) kube-proxy, and the container runtime. And because Kubernetes is highly extensible, we also get addons such as DNS, a Web UI, container resource monitoring, and cluster-level logging.

**Confused by:**
- Self-healing mechanism (weakest): it is controllers running control loops. The ReplicaSet controller (managed by a Deployment) in the kube-controller-manager reconciles actual replica count to desired; on node failure the node controller marks pods gone, the ReplicaSet controller creates a replacement, the scheduler places it, the kubelet runs it. Self-healing = the desired-state loop doing its normal job.
- Not a PaaS: because Kubernetes operates at the container level and omits opinionated app-level services (no built-in DB/middleware/build pipeline, no dictated framework), not because it "doesn't host anything" (it does run the containers).
- Two-pointer signal: it is valid when each pointer move eliminates part of the search space without backtracking. Two flavors: sorted/monotonic data (Two Sum II, 3Sum) or comparing from both ends of a symmetric structure (Valid Palindrome). Nuance worth keeping: in Two Sum II the order is a KNOWN precondition I exploit, whereas in Valid Palindrome the symmetry is the property I am TESTING, not something known in advance. So the robust signal is the no-backtracking elimination itself, not just "the array is sorted."
- 3Sum: nums[i] is the fixed FIRST element (not the middle), and the complexity is O(n^2) (sort O(n log n) dominated by the n iterations each running an O(n) two-pointer).
