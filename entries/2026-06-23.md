# 2026-06-23 — Docker Deep Dive ch.1–5 + flashcard round

**Learned:**
- Docker engine = CLI + daemon (dockerd); under the daemon sits containerd → shim → runc.
- `docker run` flow: CLI → daemon → (pull image if needed) → containerd → shim → runc creates container, then runc exits while the shim stays as parent.
- Image = immutable read-only layers; container = running instance + thin writable layer. Images include OS *userland* but NOT a kernel (containers share the host kernel — this is the key VM-vs-container distinction).
- Dockerfile order matters for layer caching: least-changing on top, most-changing (app code) on bottom.

**Confused by (mend list from flashcard round):**
1. PRIORITY — Persistence = **volumes** (Docker-managed storage outside the writable layer). I defaulted to "API calls / docker commit" which are wrong/antipatterns. Need to learn volumes vs bind mounts vs tmpfs.
2. Cache direction — I had it backwards. Least-changing on TOP, most-changing on BOTTOM. `Using cache` means a layer did NOT change. Invalidation cascades downward.
3. Registry hierarchy — Registry ⊃ Repository ⊃ tagged image. Docker Hub IS a registry (the default), not a collection of registries.
4. Compose — it's for multi-SERVICE apps defined in one YAML, with auto-networking between services (reach by service name). Not mainly about "same env for many people."

**Flashcard round score:** ~6.75/10 avg. Strong on architecture/run-flow (9/10), gaps on persistence and caching direction.
