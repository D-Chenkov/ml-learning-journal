- Gradient accumulation, `backward()` adds into `.grad` (`+=`), it does not overwrite. This is intentional and useful (the actual technique behind gradient accumulation: run several forward/backward passes without `zero_grad()`, then one `step()`, to simulate a larger effective batch size than fits in memory). Done by accident across ordinary training iterations, it's a correctness bug, not a memory bug: the accumulated gradient reflects multiple batches' worth of signal instead of one, inflating the update magnitude, which can overshoot minima and cause the loss to diverge instead of converge.
- `optimizer.zero_grad()` must be called before `loss.backward()` each iteration (not after): calling it after `backward()` but before `step()` wipes the gradients before the optimizer can use them, so `step()` has nothing to work with.
- Loss vs. optimizer, division of responsibility, corrected mid-round: loss computes the error (forward pass) and, via `backward()`, produces the gradients. The optimizer never touches the loss value itself, it only reads each parameter's `.grad` (already populated by `backward()`) and updates weights using that plus its own hyperparameters (learning rate, momentum, etc.). "Planner vs. executor" is a fine mental model as long as the executor is understood to only ever see the gradients, not the loss number.
- Device management: model and every tensor it operates on (inputs, and target/label tensors going into the loss) must all be moved to the same device (`.to(device)`) explicitly, PyTorch does not implicitly cast across devices. Mismatched devices (e.g. model on GPU, input still on CPU) raise a `RuntimeError` at the first operation that needs to combine them (e.g. the first `Linear` layer's matmul), not a silent slowdown or a CPU/GPU-concurrency issue.

**Confused by:**
- First explanation of skipping `zero_grad()` blamed it on the gradient tensor "growing too big for your computer to handle." That's wrong on the mechanism, gradient tensors stay exactly the same shape as their parameters regardless of accumulation, nothing grows in size. The actual failure is a wrong-magnitude update (stale/summed gradient signal from prior batches folded into the current step), a correctness problem, not a memory problem. Corrected after being asked to restate the PyTorch-specific mechanism (`+=` accumulation) rather than a vague consequence.
- Said the optimizer "grabs the loss and gradients" to update weights. In fact it only ever reads `.grad`; the loss scalar's role ends once `backward()` differentiates it into gradients. Small but real distinction, the optimizer has no notion of what the loss value was, only the derivative information already stored on each parameter.
- Initially described the CPU/GPU device mismatch error as "trying to work with the CPU and GPU at the same time in a way that shouldn't be possible," which is close but a little vague about mechanism. More precisely: PyTorch doesn't automatically move tensors across devices for an operation; a specific op (the first one needing both operands) finds mismatched devices and raises immediately, it's not a general "CPU and GPU fighting," it's one explicit operator-level compatibility check.
