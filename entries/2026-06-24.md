# 2026-06-24 — Docker ch.4 reread + Arrays & Hashing problems

**Learned:**
- Re-grounded the Docker engine flow (ch.4): `docker run` → daemon → containerd → shim → runc; runc creates the container then exits, shim stays as parent.
- Encode/Decode strings via a hex approach: `txt.encode('utf-8').hex()` joined with `$`. `$` is a provably safe delimiter because `.hex()` only emits the 16-symbol alphabet `0-9a-f`, so the delimiter can never collide with the payload.
- Debugged my own encode/decode: the intermediate representation is **bytes, not text**. Inverse pairs are `.encode('utf-8')`↔`.decode('utf-8')` and `.hex()`↔`bytes.fromhex()`; I had been re-encoding on both ends.
- Hash map is the spine of all of today's problems, in distinct roles: frequency counting (`Counter`), canonical-key grouping (Group Anagrams), and value→index lookup (Two Sum).

**Confused by:**
- PRIORITY — Docker **shim** + daemonless containers. The shim's two jobs are (1) keep stdio streams (STDIN/OUT/ERR) open and (2) report the container's exit status to the daemon. Because the shim — not `dockerd` — parents the container, the daemon can be restarted/upgraded **without killing running containers** (zero-downtime engine upgrades). I had the mechanism wrong.
- PRIORITY — **Bucket sort precondition** (2nd time this slipped). Frequencies are bounded by `len(nums)`, so you can index `n+1` buckets by frequency and read high→low → O(n), beating any comparison sort. The enabling property is *bounded integers*, not just "numerical."
- `most_common(k)` uses a **heap** internally (`heapq.nlargest`), O(m log k) where m = distinct values — not a tuple, and not O(k).
- Hashability vocabulary: dict keys must be hashable; list = mutable = unhashable, tuple/str = immutable = hashable. So `tuple(count)` as a key, never the raw list.
- Mixed up which problem uses the value→index trick — that's Two Sum, not Group Anagrams.
