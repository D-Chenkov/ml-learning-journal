**Learned:**
- A RAG system has 3 core components: Question (Q), Retrieved Context (C), Answer (A). Their conditional relationships ("quality of X given Y") give exactly 6 evals, no hidden variables.
- Tier 1 (foundation, before RAG eval): Retrieval Precision & Recall. Fast, no LLM needed, measures how well the retriever finds relevant documents.
- Tier 2 (Primary): Context Relevance (C|Q), Faithfulness/Groundedness (A|C), Answer Relevance (A|Q).
- Tier 3 (Advanced): Context Support Coverage (C|A), Question Answerability (Q|C), Self-Containment (Q|A).
- Domain changes which eval to emphasize: medical needs higher faithfulness (A|C), customer service needs better answer relevance (A|Q), technical docs need stronger question answerability (Q|C).
- When a RAG system fails, it fails along one of these 6 relationships, so diagnosis means finding which one. Same scientific-method loop as the eval-process reading.

**Reflection (book report):**

This is less of a "good practices and soft skills" piece than the other texts we read; it is a lot more focused, so pardon the change of register. All credit to Jason Liu, "There Are Only 6 RAG Evals" (2025/05/19), https://jxnl.co/writing/2025/05/19/there-are-only-6-rag-evals/.

Paraphrasing for my own sake: he establishes that there are 3 core components to a RAG system, the Question (Q), the Retrieved Context (C), and the Answer (A). From the conditional combinations of these 3 variables we get 6 possible relationships. The author also notes the foundation underneath them all, Retrieval Precision and Recall, which lets us measure how well we retrieve documents from the corpus.

He groups the relationships into two sets. Primary RAG relationships: 1) Context Relevance (C|Q), 2) Faithfulness/Groundedness (A|C), 3) Answer Relevance (A|Q). Advanced RAG relationships: 4) Context Support Coverage (C|A), 5) Question Answerability (Q|C), 6) Self-Containment (Q|A).

Yes, the fact that they are essentially the same pairs flipped around confused me at first too. To my best understanding it is about how each component relates to the other in a given direction. A simple way to describe it without looking back at the document: A|C (faithfulness) asks whether the answer is fully supported by the context, while C|A (context support coverage) asks whether the context actually contains everything needed to back the answer's claims. It is this directional relationship that matters when looking into RAG implementations.

I am going to quote this directly, since it will matter a lot when we start looking for jobs. The need for each relationship can vary by field:

- Medical RAG systems need higher faithfulness scores (A|C)
- Customer service RAG demands better answer relevance (A|Q)
- Technical documentation RAG requires stronger question answerability (Q|C)

So not everything is as cut and dry as wanting "high scores" on a specific relationship. The idealistic goal would be everything as high as possible, but in implementation we have to weigh the trade-offs and benefits.

In the same vein, if something is failing, we can look back at one of the 6 relationships to determine what is wrong. We are basically falling back into the idea that RAG evaluation is the scientific method once more.

That is it. This is a very short and sweet article, but definitely worth a look for the examples provided. Seeing as the metrics I was using previously covered more than 9 elements, I am left wondering if we have unintentionally been making this more complicated than it needs to be.
