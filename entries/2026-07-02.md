**Learned (Nana K8s video, part 2 - not following along hands-on; will build my own project later as reference):**

DEMO PROJECT (mongo-express + mongoDB): full app setup with K8s components. MongoDB runs as an INTERNAL service (no external requests reach it). Mongo Express Deployment needs a DB URL to connect plus DB user/password for auth; its Deployment.yaml uses env variables that reference a ConfigMap (DB URL) and a Secret (DB user/password). External URL = Node IP + external Service port.

SECRET config file: kind Secret; metadata.name; type Opaque (default for arbitrary key-value pairs); data holds the key-value pairs (base64).

SERVICE config file: kind Service; metadata.name; selector (connects to the pod via LABEL); ports (port = service port, targetPort = container port of the deployment).

CONFIGMAP: external, centralized configuration other components can use. File: kind ConfigMap; metadata.name; data (key-value pairs).

EXTERNAL SERVICE: set type: LoadBalancer (assigns an external address, accepts external requests) plus nodePort (port for the external IP). Internal default type is ClusterIP.

NAMESPACES: a "virtual cluster inside a cluster." 4 default namespaces: kubernetes-dashboard (minikube only), kube-system (system processes, do NOT modify), kube-public (publicly accessible data, a configmap with cluster info), kube-node-lease (node heartbeats / lease objects for availability), and default (where your resources go). Create with kubectl create namespace or a config file. Why use them: (1) logical grouping of complex apps, (2) avoid team conflicts in default, (3) resource sharing across staging/dev in one cluster, (4) access control + resource limits (CPU/RAM/storage) per namespace. Each namespace needs its OWN ConfigMap/Secret; reach a service in another namespace via service-name.namespace. Some resources are not namespaced (Volume, Node) and live globally; check with kubectl api-resources --namespaced=false / =true. Change active namespace with kubens (brew install kubectx).

INGRESS: routing rules that forward requests to an internal service. The internal service Ingress talks to has NO nodePort and uses the default type ClusterIP. Host = a valid domain mapped to the node's IP (the entry point). Flow: app pod <- app service (internal) <- Ingress <- Ingress Controller pod. You NEED an Ingress Controller to evaluate/process the rules (manages redirects, is the cluster entry point; many implementations, e.g. the K8s nginx ingress controller). An Ingress resource does nothing without a controller. Environment matters: a cloud provider gives an out-of-the-box virtualized load balancer; on bare metal you must configure your own entry point / proxy server with a public IP and open ports (no server in the cluster is externally accessible by default). In minikube: minikube addons enable ingress starts the nginx controller. More use cases: multiple paths per host, multiple hosts/subdomains, and TLS (data keys must be tls.crt / tls.key, values are file CONTENTS not paths, and the Secret must be in the same namespace as the Ingress).

HELM: the package manager of Kubernetes. Packages YAML manifests into charts and distributes them via public/private repos. Charts are reusable, parameterized configs (e.g. a database or a logging/monitoring stack like Elastic), so you deploy a whole app with one command instead of hand-writing many manifests.

**Confused by:**
- Service types (new mend): ClusterIP = default, internal-only, what Ingress talks to; NodePort = exposes on each node's IP:port; LoadBalancer = external address via a cloud load balancer. I conflated "expose externally" with ConfigMap / cross-namespace access.
- Service -> pods (now CLOSED): a Service uses its SELECTOR to match pod LABELS, and load-balances across the matches. The load balancer is HOW traffic spreads; the selector is WHICH pods are in the pool. Not the scheduler, not the replica count.
- An Ingress resource is inert without an Ingress Controller installed to act on it (e.g. minikube addons enable ingress).
